# ADR-005: 永続化（SQLite）は Phase 4 まで導入しない

- **日付**: 2026-06-20
- **状態**: Superseded（Phase 4-a で実装。`src/store.rs` / `docs/data-model.md` 参照）

> この ADR の「MVP〜Phase 3 では永続化しない」という判断は当時の意図どおり機能した。
> Phase 4 で予定どおり rusqlite(bundled) による永続化・リグレッション検出を実装済み。

## 背景

評価結果の履歴管理・リグレッション検出には永続ストアが有用で、草案では SQLite を想定している。一方、MVP の主目的は「1 回の評価を一気通貫で動かす」ことであり、履歴は必須ではない。

## 決定

MVP〜Phase 3 では永続化しない。結果は標準出力または Markdown ファイルとして出力する。SQLite（`rusqlite`）による永続化は Phase 4 で導入する。

## 理由

- 永続化を入れるとスキーマ設計・マイグレーション・I/O 層が増え、パイプライン確立が遅れる。
- Markdown/JSON 出力があれば、当面の比較・共有はファイルで足りる。
- スキーマは評価軸が固まる Phase 3 以降の方が安定して設計できる（手戻り回避）。

## トレードオフ

- 過去 run との自動差分比較は Phase 4 まで使えない。
- 結果はファイル管理に依存するため、ユーザ側での散逸リスクがある。
